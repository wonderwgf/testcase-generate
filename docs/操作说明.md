## 1. 工具概述

本工具通过 Cursor AI Agent 自动完成从需求文档到测试用例的完整生成流程，包括：

- 需求文档自动转换（Word/Excel → Markdown）
- 需求解析与风险识别
- 测试概要生成
- 测试用例生成
- XMind 思维导图导出

**核心特性**：
- **多项目隔离**：工作区下可并行存在多个项目（如 `acflow`、`scpr`、`wonder`），互不干扰
- **多迭代管理**：同一项目下按迭代子目录组织，历史产出完整保留
- **断点续跑**：基于文件系统判断进度，新开会话自动从上次中断处继续，无需记忆执行到哪步
- **大文档支持**：超过 800 行的需求文档自动拆分为模块批量处理，支持跨会话逐批次完成

## 2. 快速开始

### 第一次使用（全新项目）

**步骤 1：初始化项目目录**

```
/testcasegen <项目名> 初始化
```

例：`/testcasegen acflow 初始化`

这会在工作区根下创建标准目录结构（详见第 3 节）。

**步骤 2：放入需求文档**

将本迭代的需求文档（`.docx` / `.xlsx`）放入：

```
<项目>/input/prd/<迭代标识>/
```

例：`acflow/input/prd/V4.5/需求规格说明书.docx`

**步骤 3：启动全流程生成**

```
/testcasegen acflow V4.5
```

Agent 会自动按顺序完成所有步骤，在需要人工确认的节点暂停等待。

---

### 已有项目（新迭代）

直接放入新迭代的需求文档到对应子目录，然后运行：

```
/testcasegen <项目名> <新迭代标识>
```

## 3. 项目目录结构

```
<项目>/
├── input/
│   ├── prd/                              ← 需求文档输入目录
│   │   ├── <迭代1>/                      ← 迭代专属需求（docx/xlsx）
│   │   │   └── 需求规格说明书.docx
│   │   ├── <迭代2>/
│   │   └── 通用业务规则清单.xlsx          ← 根下文件 = 跨迭代共享辅助文档
│   │
│   └── knowledge/                        ← 知识库（跨迭代共享，要求 md 格式）
│       ├── baseline_cases/               ← 基线测试用例
│       ├── business/                     ← 业务规则 / 流程文档
│       ├── codedesign/                   ← 技术设计文档
│       └── prd/                          ← 历史迭代需求文档
│
└── output/
    ├── prd/
    │   ├── <迭代1>/                       ← 转换后的 Markdown 需求文档
    │   │   ├── 需求规格说明书.md
    │   │   └── modules/                  ← 大文档拆分产出（800行以上时生成）
    │   └── 通用业务规则清单.md            ← 共享文档转换产出
    ├── prd_analysis/                      ← 需求解析报告
    │   └── <迭代>_需求解析报告.md
    ├── test_outline/                      ← 测试概要
    │   └── <迭代>_测试概要.md
    ├── test_cases/                        ← 测试用例
    │   └── <迭代>_测试用例.md
    ├── modules/                           ← 大文档模块循环中间产出
    │   └── <迭代>/
    │       ├── 01/
    │       │   ├── prd_analysis.md
    │       │   ├── test_outline.md
    │       │   └── test_cases.md
    │       └── 02/
    └── xmind/                            ← XMind 导出
        └── <迭代>_测试用例.xmind
```

**关键规则**：
- `input/prd/<迭代>/` 下的文件 = 本迭代专属需求
- `input/prd/` 根下（不在任何子目录）的文件 = 跨迭代共享辅助文档（自动纳入每次分析）
- `input/knowledge/` 下的文件跨所有迭代共享，统一作为知识库参考

## 4. 完整工作流说明

工作流分为三个阶段，共 6 个主要步骤：

```
Phase 1（全自动）   →   Phase 2（AI生成，有节点暂停）   →   Phase 3（全自动）
Step 1/1.5/2/2.5/2.7      Step 3/4/5                         Step 6
```

### Phase 1 · 环境准备（全自动执行）

#### Step 1 · 初始化目录

检查并创建项目目录结构。已存在则跳过。

#### Step 1.5 · 输入文件检查

- 扫描 `input/prd/<迭代>/` 是否有文件
- 若目录为空，暂停提示放入文档，用户说「继续」后从 Step 2 重新开始
- 若 `input/knowledge/` 为空，给出一次性提示（不阻断流程）

#### Step 2 · 文档格式转换

将 `input/prd/` 下所有 `.docx` / `.xlsx` 文件转换为 Markdown：

- 迭代目录下的文件 → `output/prd/<迭代>/`
- 根级共享文档 → `output/prd/`
- 已转换的文件自动跳过（幂等）

#### Step 2.5 · 知识库索引

扫描 `input/knowledge/` 下的 md 文件，自动生成/刷新 `knowledge_index.md`：

- 知识库有变更（新增/修改/删除文件）→ 自动刷新索引
- 已是最新 → 跳过

Step 3/4/5 通过索引精准引用知识库内容，而非全量读取，避免超出上下文限制。

#### Step 2.7 · 大文档检测与拆分

扫描 `output/prd/<迭代>/` 下的 md 文件：

- **总行数 ≤ 800 行** → 跳过，走普通模式
- **总行数 > 800 行** → 触发拆分，进入**模块循环模式**（详见第 6 节）

---

### Phase 2 · AI 生成（关键阶段）

#### Step 3 · 需求解析

读取需求 md 文档，生成**需求解析报告**，内容包含：

- 功能模块拆解（MTS 模块）
- 业务规则与约束梳理
- 需求不完备 / 测试风险点汇总

产出：`output/prd_analysis/<迭代>_需求解析报告.md`

**暂停条件**：发现风险点 > 0，或大文档预算触顶。暂停后列出所有风险点，等待用户确认是否需要补充材料。

#### Step 4 · 测试概要

基于需求解析报告，生成**测试概要**，内容包含：

- 各功能模块的测试点覆盖清单
- 测试策略（正向/异常/边界/非功能）

产出：`output/test_outline/<迭代>_测试概要.md`

**暂停条件**：发现信息源冲突或关键缺口时暂停，或大文档预算触顶。

#### Step 5 · 测试用例

基于测试概要，生成完整**测试用例**，格式为 Markdown 表格：

| 字段 | 说明 |
|------|------|
| 模块 | 功能模块名称 |
| 用例标题 | 简洁描述测试点 |
| 前置条件 | 执行前的系统状态 |
| 测试步骤 | 操作步骤 |
| 预期结果 | 期望的系统行为 |
| 用例类型 | 正向/异常/边界/非功能 |

产出：`output/test_cases/<迭代>_测试用例.md`

**暂停条件**：大文档预算触顶时暂停。

---

### Phase 3 · 导出（全自动执行）

#### Step 6 · 合并 + 生成 XMind

- **模块循环模式**：先合并所有模块的产出（需求解析报告、测试概要、测试用例），再导出 XMind
- **普通模式**：直接导出 XMind

产出：`output/xmind/<迭代>_测试用例.xmind`

---

### 步间自动/暂停规则

| 步骤 | 自动继续条件 | 暂停条件（任一满足即暂停） |
|------|------------|--------------------------|
| Step 3 完成后 | 全量完成 + 风险点为 0 + 预算未触顶 | 有风险点 / 未全量完成 / 预算触顶 |
| Step 4 完成后 | 全量完成 + 无冲突/缺口 + 预算未触顶 | 有冲突或缺口 / 未全量完成 / 预算触顶 |
| Step 5 完成后 | 全量完成 + 预算未触顶 | 未全量完成 / 预算触顶 |
| Step 6 | 始终自动执行，完成后汇报 | — |

## 5. 命令参数速查

### 基础命令格式

```
/testcasegen <项目名> [迭代标识] [关键词或参数]
```

### 参数说明

| 参数/关键词 | 说明 | 示例 |
|------------|------|------|
| `--from <步骤>` | 从指定步骤开始，执行后续全部步骤 | `--from 4` |
| `--only <步骤>` | 只执行该步骤，不继续向后 | `--only 2.5` |
| `--iter <标识>` | 显式指定迭代标识 | `--iter V4.5` |

### 关键词快捷方式

| 关键词 | 等价于 | 说明 |
|--------|--------|------|
| `初始化` | `--only 1` | 只初始化目录结构 |
| `文档转换` / `转md` | `--from 2` | 从文档转换开始（加「只」→ `--only 2`） |
| `知识库索引` / `只刷索引` | `--only 2.5` | 只刷新知识库索引 |
| `需求解析` | `--from 3` | 从需求解析开始 |
| `测试概要` | `--from 4` | 从测试概要开始 |
| `测试用例` / `生成用例` | `--from 5` | 从测试用例开始 |
| `xmind` / `导出` | `--only 6` | 只导出 XMind |

### 命令示例

```bash
# 全流程（最常用）
/testcasegen acflow V4.5

# 从测试概要开始（需求解析已完成的情况下）
/testcasegen acflow V4.5 --from 4

# 只做文档转换
/testcasegen acflow V4.5 只文档转换

# 只刷新知识库索引
/testcasegen acflow --only 2.5

# 只导出 XMind
/testcasegen acflow V4.5 导出

# 用 @ 方式指定文件（自动推断项目和迭代）
/testcasegen @acflow/input/prd/V4.5/ 需求解析

# 未指定迭代（Agent 自动检测或询问）
/testcasegen acflow
```

## 6. 大文档处理（模块循环模式）

当需求 md 文档总行数超过 **800 行**时，自动触发模块循环模式。

### 触发与拆分

Step 2.7 自动检测文档大小，触发后：

1. 自动运行拆分脚本，识别共享前置章节（术语/概述/参考文档等）
2. 按功能章节分组，目标每模块约 600 行（含共享前置）
3. 向用户展示拆分方案并**等待确认**：

```
📋 文档共 2400 行，建议拆分为 4 个模块：

  共享前置（每模块均包含）：术语定义、系统概述（共 120 行）

  模块 01：用户认证 + 权限管理（功能 580 行，含共享共 700 行）
  模块 02：工单管理 + 工单详情（功能 600 行，含共享共 720 行）
  模块 03：报表查询 + 数据导出（功能 580 行，含共享共 700 行）
  模块 04：系统配置 + 消息通知（功能 520 行，含共享共 640 行）

若有跨模块逻辑关联，请说明调整意见；说「确认」即按此方案继续。
```

用户说「确认」后进入模块循环处理。

### 上下文预算管理

模块循环时，按总模块数自动控制每步连续处理上限，避免上下文过长导致质量下降：

| 总模块数 | 每步连续处理上限 |
|---------|----------------|
| ≤ 6 个 | 全量处理 |
| 7 ~ 12 个 | 每步最多 6 个 |
| > 12 个 | 每步最多 4 个 |

触顶时自动暂停，输出：

```
⏸️ 上下文预算检查：本步骤已连续处理 4/19 个模块（当前档位上限 = 4），
建议新开会话继续。
请新开会话执行 /testcasegen acflow V4.5 即可自动从模块 05 恢复。
```

### 最终合并

所有模块的 Step 3/4/5 均完成后，Step 6 自动合并为完整的三份报告，再导出 XMind。

## 7. 知识库管理

知识库位于 `<项目>/input/knowledge/`，跨所有迭代共享。

### 子目录说明

| 子目录 | 存放内容 | 格式要求 |
|--------|---------|---------|
| `baseline_cases/` | 历史版本的基线测试用例 | Markdown |
| `business/` | 业务规则文档、流程说明 | Markdown |
| `codedesign/` | 技术设计文档、接口文档 | Markdown |
| `prd/` | 历史迭代的需求文档 | Markdown |

> **格式要求**：知识库文件必须是 `.md` 格式。如果原始文件是 `.docx`/`.xlsx`，请先转换：
> ```
> /testcasegen <项目> 文档转换
> ```
> 转换完成后将 md 文件移入对应子目录。

### 知识库索引（Step 2.5）

每次运行时，Agent 自动检测知识库变更并刷新索引文件 `knowledge_index.md`。Step 3/4/5 通过索引按章节范围精准读取，而非一次性加载全部文件内容，有效避免超出上下文限制。

**手动刷新索引**：

```
/testcasegen <项目> 只刷索引
```

### 使用建议

- 首次使用前，将历史测试用例（基线用例）放入 `baseline_cases/`，可显著提升生成用例的质量和格式一致性
- 将当前迭代相关的技术设计文档放入 `codedesign/`，有助于识别接口级测试点
- 知识库是跨迭代共享的，无需每次迭代重复放置

## 8. 跨会话接力

大型项目（模块数多）通常需要多次会话完成 Step 3/4/5。每次新开会话执行相同命令即可自动续接：

```
/testcasegen <项目> <迭代>
```

Agent 通过扫描 `output/` 目录下的文件存在情况，自动判断进度并从上次中断处继续，**无需用户记忆执行到哪一步**。

### 进度判断逻辑（简化说明）

| 已有产出文件 | 下次启动时的起跑步骤 |
|------------|-------------------|
| 无任何产出 | Step 1（初始化） |
| 有需求 md，无解析报告 | Step 3（需求解析） |
| 有解析报告，无测试概要 | Step 4（测试概要） |
| 有测试概要，无测试用例 | Step 5（测试用例） |
| 有完整测试用例 | 询问：补充用例 or 只导出 XMind |
| 模块循环模式，部分模块已完成 | 从第一个未完成模块继续 |

### 场景 1：全新项目，全流程生成

```

## 10. 注意事项与常见问题

### 路径与编码

- 所有路径相对于项目目录（工作区根下的项目目录，如 `acflow/`）
- 中文目录/文件名完全支持，工具内部统一使用 UTF-8 处理
- Windows 下 PowerShell 建议在 Cursor 终端中运行，避免编码问题

### 知识库必须是 md 格式

`input/knowledge/` 下的文件只能是 `.md`。如果是 Word/Excel 文件，先运行文档转换，再手动将产出的 md 文件移入对应子目录。

### 需求文档按迭代子目录组织

`input/prd/` 根下（没有放在子目录中的）文件会被识别为**跨迭代共享辅助文档**（如通用业务规则），每次迭代分析时均会自动纳入参考，无需重复放置。各迭代专属需求必须放在对应子目录中。

### 大文档拆分是幂等的

`_manifest.json` 生成后，再次运行不会重复拆分，直接复用上次方案。如果需要重新拆分（如需求文档有重大更新），删除 `output/prd/<迭代>/modules/` 整个目录后重新运行。

### 已完成的步骤自动跳过

每个步骤在开始前会检查产出是否已存在，已存在则跳过。如需重新生成某步骤的产出，手动删除对应文件后重新运行即可。

### 跨模块关联的处理

如果两个模块之间有强关联（如状态机与详情页），在 Step 2.7 拆分方案展示时提出，Agent 会按意见合并到同一模块，保证上下文完整性。

### 共享辅助文档的管理

跨迭代共享文档（如通用业务规则清单）放在 `input/prd/` 根下即可，每次迭代分析时 Agent 自动将其纳入参考，无需重复放置到每个迭代子目录。

以下参数在 `.cursor/commands/testcasegen.md` 的「配置常量」一节中定义，如有调整需同步修改对应文件。

| 参数 | 默认值 | 含义 |
