# 测试用例生成 Command 工具 — 技术专利挖掘书（模版稿）

以下正文可直接填入贵司专利挖掘表单；表述聚焦**可挖掘的技术点**（流程编排、异构文档统一、规则化多阶段生成、可恢复执行与产物规范化），与实现载体 [`.cursor/commands/testcasegen.md`](../.cursor/commands/testcasegen.md) 一致。

---

## 1、技术背景

（约 70 字）

软件交付与回归测试中，需求与设计材料常以 Word、Excel 等异构格式分散存放，测试分析与用例编写高度依赖人工通读与经验，环节多、重复劳动大。企业希望在智能开发助手环境中，将**文档预处理、测试设计、用例落盘与评审交付**串联为可重复执行的规范化流水线。

---

## 2、技术问题

（约 280 字）

现有做法普遍存在以下问题：**其一**，PRD 与设计说明格式不统一，测试人员需在多工具间切换阅读，难以稳定抽取规则、枚举与界面细节，导致用例覆盖不均衡或关键约束遗漏。**其二**，从“读需求”到“出测试概要”再到“细化为用例”缺乏统一方法论约束，输出风格与目录结构因人而异，不利于版本对比与复用。**其三**，长文档或大型项目单次生成易超上下文窗口，流程无法从中间步骤恢复或单步重跑，增量变更成本高。**其四**，评审与协作常需思维导图等形态，若与 Markdown 用例割裂，需二次手工整理。**其五**，缺少与工程目录绑定的约定（input/output、命名前缀），自动化脚本与 AI 步骤难以在同一项目内稳定协同。

因此需要一种**可编排、可分段执行、输入输出路径明确**的测试资产生成方案，在保证可追溯（规则文件驱动）的前提下，降低人工搬运与格式转换成本。

---

## 3、技术方案

（约 320 字）

采用**单入口命令 + 分阶段流水线**的技术手段：以 Cursor Command 触发全流程，按阶段自动或交互执行。

**环境准备阶段**：检测项目是否具备约定目录结构；若无则通过初始化技能创建 `input/`、`output/` 等骨架；扫描 `input/prd/`、`input/codedesign/` 下未转换为 Markdown 的 Office 文档，按需调用批量转换技能，使后续 AI 步骤统一在 Markdown 语料上工作。

**AI 生成阶段**：依次加载三套结构化规则（需求解析、测试概要、测试用例），并组合读取已转换的需求/设计文档及可选知识库；每步产出写入约定子目录（如 `prd_analysis`、`test_outline`、`test_cases`），形成**解析 → 概要 → 用例**的递进式中间件，便于人工校对与断点续做。

**导出阶段**：将最新测试用例 Markdown 自动转为 XMind，便于评审与分享。

**执行控制**：支持 `--from`、`--only` 等参数，从指定步骤启动或仅执行单步，适配大文档分模块处理与增量迭代。

整体上，通过**目录约定、异构文档归一、规则驱动的多阶段生成与可选导图导出**，实现测试用例从需求到交付形态的一体化、可恢复生成。

## 可进一步挖掘的权利化角度（供内部讨论，非模版必填）

- **分阶段规则注入 + 固定目录产物** 是否构成区别于“单次提示词生成用例”的方法特征。
- **Office→MD 增量检测（同名 md 是否存在）** 作为预处理子流程的技术效果（减少重复转换、保证 AI 输入一致）。
